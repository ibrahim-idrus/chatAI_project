## Commands

All commands run from the repo root unless otherwise noted. This is an npm workspaces monorepo (`apps/*`, `packages/*`).

| Task | Command |
|---|---|
| Run both apps in dev | `npm run dev` (backend on `127.0.0.1:8787`, frontend on `5173`) |
| First-time / after schema change | `npm run dev:setup` (runs `db:push` then `dev`) |
| Frontend only | `npm run dev:frontend` (Vite) |
| Backend only | `npm run dev:backend` (`wrangler dev`) |
| Build all packages + apps | `npm run build` (in dependency order: db → types → utils → frontend → backend) |
| Lint | `npm run lint` (root) or `npm run lint -w @chatai/frontend` |
| Frontend typecheck | `npm run build -w @chatai/frontend` (runs `tsc -b && vite build`) |
| Drizzle: generate migration | `npm run db:generate` |
| Drizzle: apply migrations | `npm run db:migrate` |
| Drizzle: push schema (dev) | `npm run db:push` |
| Drop all tables (dev reset) | `tsx packages/db/scripts/reset.ts` |
| Backend deploy | `npm run deploy -w @chatai/backend` (wrangler) |

`DATABASE_URL` is loaded from `packages/db/.env` first, then repo-root `.env` (see `packages/db/drizzle.config.ts`). Backend secrets (`OPENAI_API_KEY`, `JWT_SECRET`, `DATABASE_URL`) come from Cloudflare bindings / `.dev.vars` for local Wrangler.

## Architecture

### Edge-first stack
Backend is a **Cloudflare Worker** (Hono + Wrangler) talking to **Neon Postgres** via the HTTP driver — no long-lived connections, so `getDb(env)` creates a fresh Drizzle client per request. Frontend is a Vite SPA that proxies `/api/*` to the worker in dev (`apps/frontend/vite.config.ts`).

Three Cloudflare bindings, all defined in `apps/backend/wrangler.toml` and typed in `apps/backend/src/env.ts`:
- `SESSION_KV` — KV namespace, source of truth for active sessions (TTL = 7d).
- `CHAT_RETRY_QUEUE` — Cloudflare Queue. Producer + consumer in the same worker; the consumer is exported as `queue: handleRetry` from `src/index.ts`.
- `AI` — Workers AI binding (declared but currently unused; OpenAI is called directly via `OPENAI_API_KEY`).

### Auth model (three-layer session)
On login (`apps/backend/src/routes/auth.ts`):
1. JWT signed with `JWT_SECRET` (jose, HS256, 7d) carries `{ sub, sessionId, role }` and is set as an **HttpOnly, Secure, SameSite=Strict cookie** named `token`.
2. The `sessionId` is stored verbatim in `SESSION_KV` with `expirationTtl`.
3. A SHA-256 hash of `sessionId` is stored in the `sessions` table (the raw id never hits Postgres).

`authMiddleware` (`apps/backend/src/middleware/auth.ts`) requires **all three** to be valid: cookie present, JWT verifies, and KV record exists. Logout deletes from KV + DB and clears the cookie. Frontend hydrates auth in `routes/__root.tsx` via `beforeLoad` calling `getMe()` and populating the `useAuthStore` Zustand store; the `_auth` layout route then gates protected routes by reading that store synchronously.

When changing auth, the three layers must stay consistent — adding a session step that only writes to KV but not the DB (or vice versa) will silently desync.

### Streaming + retry pipeline
`POST /api/chat/stream` (`apps/backend/src/routes/chat.ts`):
1. Insert user message (`status: completed`).
2. Insert empty assistant placeholder (`status: pending`).
3. Build chat history from `completed` messages only.
4. Call `streamAIResponse` (`src/lib/stream.ts`) — Vercel AI SDK `streamText` over OpenAI, returns a `Response` from `toTextStreamResponse()`.
5. `onFinish` updates the assistant message, inserts `token_usage`, increments `threads.totalTokens`.
6. **If streaming throws**, push a `RetryPayload` to `CHAT_RETRY_QUEUE` and return 503. The queue consumer (`src/lib/retry.ts`) calls non-streaming `generateText`, updates the same `messageId`, and marks `status: failed` after 3 attempts.

The frontend has **two** chat implementations — keep this in mind when editing:
- `apps/frontend/src/hooks/useThreadChat.ts` — uses `@ai-sdk/react`'s `useChat` with `TextStreamChatTransport`.
- `apps/frontend/src/routes/_auth/chat.tsx` — implements its own `fetch` + `ReadableStream` reader and manages messages in local `useState`. This is the route currently rendered.

### Routing (frontend)
TanStack Router with file-based routes in `apps/frontend/src/routes/`:
- `__root.tsx` — root, hydrates auth.
- `_auth.tsx` — pathless layout that redirects to `/login` if no user.
- `_auth/chat.tsx`, `_auth/profile.tsx`, `_auth/HistoryPage.tsx` — protected.
- `login.tsx`, `register.tsx`, `index.tsx` — public.

`routeTree.gen.ts` is generated by the `tanstackRouter` Vite plugin and is **gitignored** — don't edit it. Path alias `@/*` resolves to `src/*` (used by shadcn/ui components).

### Shared packages
- `@chatai/db` — Drizzle schema (`users`, `sessions`, `threads`, `messages`, `token_usage`, `event_logs`) and `createClient(url)`. The `main` field points at TypeScript source (`./src/index.ts`); consumers import directly without a build step.
- `@chatai/types` — Zod schemas for auth + chat, shared between frontend and backend. Edit these here, not in app-local copies.
- `@chatai/utils` — small helpers (`slugGen`, `tokenCount`).

## Conventions

- **Comments and error messages are often in Indonesian** (e.g. `'Email atau password salah'`, `'Session expired, silakan login ulang'`). Match the surrounding language when editing user-facing strings.
- Validation uses Zod everywhere — `@hono/zod-validator` on the backend, `@hookform/resolvers/zod` on the frontend, with shared schemas in `@chatai/types` where possible.
- Event logging: call `logEvent(db, { eventType, userId?, payload?, ip? })` from `src/lib/logger.ts` for any auth/thread/message lifecycle event. It's fire-and-forget (no `await`) to keep request latency down.
- All DB writes use Drizzle's typed query builder. Raw SQL only via `sql\`...\`` for things like the `count(*)::int` aggregate or `totalTokens + ${n}` increments.
